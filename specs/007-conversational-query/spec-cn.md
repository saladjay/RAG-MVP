### 用户故事 1 - 对话式信息收集 (优先级: P1)

用户以模糊查询开始，比如"我想找关于安全的规定"。系统不会立即搜索，而是通过对话收集缺失的信息：

1. 系统: "请问您需要查找哪一年的安全规定？"
2. 用户: "去年的"
3. 系统: "2024年有以下安全相关的规定：安全生产管理办法、消防管理规定、应急预案等。请问您具体需要哪方面的？"
4. 用户: "消防相关的"
5. 系统: "好的，让我为您搜索2024年消防相关的规定文件。"

对话历史被保留，最终搜索包含所有收集的上下文。

**为什么是这个优先级**: 这是核心能力 - 多轮对话通过自然对话逐步收集信息，显著提高查询质量。

**独立测试**: 可以通过发起不完整查询并验证系统正确提出后续问题并构建上下文来进行完全测试。

**验收场景**:

1. **给定** 用户提交 "我想找关于安全的规定"，**当** 系统处理查询时，**则** 应该识别缺失的年份和具体主题维度并启动对话流程
2. **给定** 系统询问年份且用户响应 "去年的"，**当** 系统解释响应时，**则** 应该将 "去年的" 转换为 "2024年" 并继续收集信息
3. **给定** 系统已收集年份和主题信息，**当** 继续搜索时，**则** 最终查询应该包含 "2024年" 和 "消防相关" 上下文
4. **给定** 用户在多轮中提供不一致的信息，**当** 系统检测到不一致时，**则** 应该请求澄清

### 用户故事 2 - 口语表达识别 (优先级: P1)

用户使用与官方文档术语不同的非正式或口语化语言：

- 用户说: "有没有关于防火的规定"
- 系统识别: "防火" → "消防"（防火 → 消防）
- 用户说: "我要找去年发的关于大家加班的文件"
- 系统识别: "加班" → "劳动节假日安排" 或 "值班工作"

系统维护口语术语到正式文档术语的映射，并使用它来理解用户意图。

**为什么是这个优先级**: 用户自然使用非正式语言。将口语表达映射到正式术语对于准确检索至关重要。

**独立测试**: 可以通过提交带有口语术语的查询并验证系统正确将它们转换为正式术语来进行测试。

**验收场景**:

1. **给定** 用户提交 "有没有关于防火的规定"，**当** 系统处理查询时，**则** 应该将 "防火" 映射到 "消防" 并搜索消防规定
2. **给定** 用户提交 "我要找关于开会的文件"，**当** 系统处理查询时，**则** 应该映射到 "会议通知" 或 "会议纪要"
3. **给定** 用户提交 "有没有关于涨工资的文件"，**当** 系统处理查询时，**则** 应该映射到相关术语，如 "超额利润分享" 或 "薪酬福利"
4. **给定** 用户提交包含未映射口语术语的查询，**当** 系统处理查询时，**则** 应该尝试找到相似术语或请求澄清

### 用户故事 3 - 同义词和相关词扩展 (优先级: P2)

当用户搜索某个概念时，系统自动使用相关词扩展查询以改善召回率：

- 用户说: "关于党建的文件"
- 系统扩展包括: "党风廉政"、"党纪学习教育"、"党员大会" 等

系统使用基于文档内容分析的相关概念知识图来增强搜索全面性。

**为什么是这个优先级**: 同义词扩展通过找到使用不同术语表示相同概念的文档来改善搜索召回率。

**独立测试**: 可以通过提交查询并验证相关词包含在搜索中来进行测试。

**验收场景**:

1. **给定** 用户提交 "关于党建的文件"，**当** 系统处理查询时，**则** 应该扩展搜索以包括相关术语，如 "党风廉政"、"党纪学习教育"、"党员大会"
2. **给定** 用户提交 "关于人事的文件"，**当** 系统处理查询时，**则** 应该扩展以包括 "任免"、"选举"、"评优"
3. **给定** 用户提交 "关于安全的文件"，**当** 系统处理查询时，**则** 应该扩展以包括 "安全生产"、"消防"、"应急预案"、"防汛"
4. **给定** 用户提交不需要扩展的具体查询，**当** 系统处理查询时，**则** 不应该过度扩展并稀释搜索结果

### 用户故事 4 - 对话上下文持久化 (优先级: P1)

用户进行跨越几分钟的多轮对话：

1. 用户: "我想找关于安全生产的文件"
2. 系统: "请问您需要查找哪一年的安全生产文件？"
3. 用户: "2024年的"
4. [用户在另一个会话中询问其他内容]
5. 用户: "还有关于消防的呢？"
6. 系统: 记住之前的上下文并搜索 "2024年消防相关" 文件

对话上下文在多轮中保持，允许系统引用之前收集的信息。

**为什么是这个优先级**: 上下文持久化是自然多轮对话的基础。没有它，每一轮都是独立的，对话流程会被打断。

**独立测试**: 可以通过进行多轮对话并验证上下文在多轮中保持来进行测试。

**验收场景**:

1. **给定** 用户在之前的轮中建立了 "2024年" 上下文，**当** 用户在不指定年份的情况下提出后续问题时，**则** 系统应该使用之前建立的年份上下文
2. **给定** 用户建立了 "工会" 上下文，**当** 用户问 "有没有关于开会的"，**则** 系统应该解释为 "工会会议"
3. **给定** 对话跨越多个会话，**当** 用户在一段不活动后返回时，**则** 系统应该选择性地总结之前的上下文或根据用户偏好重新开始
4. **给定** 用户明确开始新主题，**当** 系统检测到主题变化时，**则** 应该清除无关上下文，同时保留相关信息

### 用户故事 6 - 结构化查询提取和分类 (优先级: P1)

系统执行查询元素的结构化提取并分类查询类型以确定适当的响应策略。每个用户输入都被分析以产生包含以下内容的结构化输出：

**提取元素**:
- **时间**: 年份（2024）、日期（完整日期）、时间范围（近三年/去年/上半年）、相对时间（今年/本月）
- **空间**: 组织（粤东科/党总支/工会）、位置类型（公司级/部门级）
- **文档**: doc_type（通知/通报/报告/请示/纪要）、doc_number（文号）
- **知识库**: company_id（公司标识，如 N000002）、file_type（文件类别: PublicDocDispatch/发文, PublicDocReceive/收文）
- **内容**: topic_keywords（主题关键词）、policy_names（完整制度名称）
- **数量**: has_count_query（"多少/几个"的true/false）、数字（明确数字）、量词（多少/几个）

**分类**:
- **主题类别**: meta_info（询问文档本身）| business_query（询问文档内容）| unknown
- **是否后续**: 检测引用先前上下文的代词（这/那/那个/它）
- **置信度**: high | medium | low，基于提取清晰度

**查询类型示例**:

*元信息查询*（询问文档本身）:
- "2024年发布了多少个制度" → year:2024, has_count_query:true, topic:制度发布
- "有没有关于安全的制度目录" → keyword:制度+目录, topic_class:meta_info
- "去年的制度有没有更新版本" → temporal:去年, keyword:版本更新

*业务查询*（询问文档内容）:
- "经理级别的住宿标准是多少" → level:经理, keyword:住宿标准, quantifier:多少
- "关于安全生产的要求是什么" → keyword:安全生产+要求
- "能不能报销交通费用" → keyword:可以报销+交通

*后续查询*:
- "上海的呢"（上下文：之前询问过北京）→ is_follow_up:true, inherit:上海
- "那消防的呢"（上下文：之前询问过安全）→ is_follow_up:true, keyword:消防

**为什么是这个优先级**: 结构化提取和分类使系统能够精确理解用户意图并将查询路由到适当的处理逻辑。这是所有其他对话能力的基础。

**独立测试**: 可以通过提交各种查询类型并验证结构化输出与预期的提取和分类匹配来进行测试。

**验收场景**:

1. **给定** 用户提交 "2024年发布了多少个制度"，**当** 系统提取和分类时，**则** 应该输出: temporal.year:[2024], quantity.has_count_query:true, entities.keywords:[制度,发布], classification.topic_class:meta_info
2. **给定** 用户提交 "经理级别的住宿标准是多少"，**当** 系统提取和分类时，**则** 应该输出: person.level:[经理], entities.keywords:[住宿,标准], quantity.quantifiers:[多少], classification.topic_class:business_query
3. **给定** 用户在询问北京后提交 "上海的呢"，**当** 系统提取和分类时，**则** 应该输出: spatial.organization:[上海], classification.is_follow_up:true, entities.keywords:inherited_from_previous_turn
4. **给定** 用户提交模糊查询，**当** 系统提取和分类时，**则** classification.confidence 应该是 "medium" 或 "low" 并触发澄清提示

### 用户故事 7 - 业务域分类和路由 (优先级: P1)

从用户查询中提取结构化元素后，系统将查询分类到业务域以实现智能路由和专门化处理：

**域分类示例**:

- 用户: "关于差旅住宿报销的规定" → business_domain: "finance", sub_domain: "accommodation"
- 用户: "有没有关于绩效考核的文件" → business_domain: "hr", sub_domain: ""
- 用户: "安全生产管理的办法" → business_domain: "safety", sub_domain: ""
- 用户: "党建工作的相关文件" → business_domain: "party", sub_domain: ""
- 用户: "档案管理规定" → business_domain: "archive", sub_domain: ""

系统使用基于优先级排序的关键字分类来确定适当的业务域，实现：

1. **专门化知识库**: 将查询路由到特定领域的文档集合
2. **域特定提示**: 为不同业务领域使用不同的提示模板
3. **专家路由**: 在需要时将复杂查询路由到领域专家
4. **分析**: 按业务域跟踪查询模式以获得洞察

**为什么是这个优先级**: 业务域分类对于将查询路由到适当的知识库和处理逻辑至关重要。它通过将搜索范围缩小到相关文档来实现专门化并提高检索准确性。

**独立测试**: 可以通过提交来自不同业务域的查询并验证正确的分类和路由来进行测试。

**验收场景**:

1. **给定** 用户提交 "关于差旅费报销的标准"，**当** 系统分类查询时，**则** 应该输出: business_domain:"finance", sub_domain:""
2. **给定** 用户提交 "住宿费能报多少"，**当** 系统分类查询时，**则** 应该输出: business_domain:"finance", sub_domain:"accommodation"
3. **给定** 用户提交 "员工绩效考核办法"，**当** 系统分类查询时，**则** 应该输出: business_domain:"hr", sub_domain:""
4. **给定** 用户提交 "安全生产应急预案"，**当** 系统分类查询时，**则** 应该输出: business_domain:"safety", sub_domain:""
5. **给定** 用户提交跨越多个域的查询，**当** 系统分类时，**则** 应该使用优先级排序并选择最高优先级域

### 用户故事 8 - 结构化查询生成用于检索 (优先级: P1)

在所有提取、分类和域路由之后，系统生成三个针对文档检索优化的独立搜索查询：

**生成输出结构**:
```json
{
  "q1": "第一个搜索问题变体",
  "q2": "第二个搜索问题变体",
  "q3": "第三个搜索问题变体",
  "must_include": "必需术语，3-6项",
  "keywords": "带同义词的扩展关键词，5-10项"
}
```

**按域生成查询**:

*业务查询域*（财务/人力/安全等）:
- 生成包含收集槽的问题：城市、费用类型、级别、主题
- 使用特定主题模板：standard（"标准是多少"）、rule（"需要哪些单据"）、process（"流程是什么"）、scope（"是否可以报销"）
- 添加位置上下文：城市名称或province_relation（省内/省外/跨省/全国）
- 扩展费用类型口语术语："住宿/酒店"、"餐饮/餐补"、"交通/打车/高铁"

*元信息域*（文档库存查询）:
- 为不同子类型生成问题：目录、计数、搜索、版本、生效日期、责任、模板、定义
- 使用域特定问题模式，而不强制使用不相关的术语，如 "报销/出差"

*未知域*:
- 对用户输入进行中性重写，并进行轻度结构化
- 避免引入原始查询中不存在的域特定术语

**示例输出**:

用户: "北京的住宿标准是多少"
```json
{
  "q1": "北京住宿费报销标准是多少",
  "q2": "北京住宿费用上限是多少",
  "q3": "北京经理级住宿标准如何规定",
  "must_include": "报销, 住宿, 标准, 北京",
  "keywords": "报销, 住宿/酒店, 标准, 上限/额度, 北京, 凭证/单据"
}
```

**为什么是这个优先级**: 这是将所有先前的分析转换为实际搜索查询的最终输出层。三问题变体方法通过捕获不同的措辞和视角来改善检索召回率。

**独立测试**: 可以通过提交查询并验证生成的输出包含适当的问题变体和关键词扩展来进行测试。

**验收场景**:

1. **给定** 用户提交 "北京的住宿标准是多少"，**当** 系统生成检索查询时，**则** 应该输出三个问题变体，其中must_include包含 "报销, 住宿, 标准, 北京"
2. **给定** 用户提交 "关于安全生产的制度有哪些"，**当** 系统生成检索查询时，**则** 应该输出关于安全生产规定的问题，而不强制使用财务术语，如 "报销"
3. **给定** 用户提交 "怎么查制度"，**当** 系统生成检索查询时，**则** 应该输出与搜索相关的问题，其中包含 "检索/搜索/关键词" 等关键词
4. **给定** 收集槽包含city="上海"且expense_type="交通"，**当** 系统生成问题时，**则** 所有三个问题都应该明确包含 "上海" 和 "交通/交通费"

**结构化提取实体**:
- **提取元素**: 表示查询解析的结构化输出，包含：

  - **内容**: topic_keywords（主题关键词列表）、policy_names（完整制度名称）
  - **数量**: has_count_query（布尔值）、numbers（明确数字列表）、quantifiers（量词列表）

- **知识库路由**: 表示文档检索的目标知识库：
  - **company_id**: 公司标识符，由外部系统提供（例如：N000002、N000003）
  - **file_type**: PublicDocDispatch（发文/发出公文）| PublicDocReceive（收文/接收公文）| both（不确定时两者都搜）

  - **collection_names**: 要搜索的知识库集合（根据确定结果选择单个或两个）

- **查询分类**: 表示路由的分类结果：
  - **topic_class**: meta_info（询问文档信息）| business_query（询问文档内容）| unknown
  - **is_follow_up**: true | false（基于是否包含指代词）
  - **confidence**: high | medium | low（基于提取明确性）

**对话管理实体**:
- **对话会话**: 表示与用户的多轮对话会话，包含对话历史、提取的上下文和当前状态

- **对话轮**: 表示对话中的单次交换，包括用户输入、系统响应、提取的元素和分类

- **信念状态**: 表示对话中积累的上下文和理解，包括：
  - history（具有提取元素的先前轮）
  - established_dimensions（时间、空间、文档、内容的确认值）

**知识实体**:
- **口语术语映射**: 表示从非正式表达映射到正式文档术语，包括口语术语、正式等价物、上下文适用性和置信度分数

- **内容类别**: 表示文档内容的类别（例如，党建工作、人事工作），具有相关术语、子类别和关键词扩展

- **查询上下文**: 表示从对话中积累的上下文，包括已建立的维度，如年份、组织、主题和文档类型

- **建议模板**: 表示常见查询模式的主动建议模板，包括触发条件、建议文本和可用选项

**路由实体**:
- **查询路由**: 表示基于topic_class确定的处理路径：
  - meta_info → 文档库存搜索和列表
  - business_query → 文档内容检索和QA
  - unknown → 澄清和信息收集

- **业务域分类**: 表示专门化路由的业务领域分类：
  - **business_domain**: finance | hr | archive | safety | governance | it | procurement | admin | party | other
  - **sub_domain**: 对于财务域 - accommodation | meal | transport | other（其他域为空字符串）
  - **confidence**: 基于关键字匹配强度和域特异性

- **生成的检索查询**: 表示文档检索的最终结构化输出：
  - **q1、q2、q3**: 搜索的三个独立问题变体
  - **must_include**: 必须出现在结果中的核心锚点术语（3-6项）
  - **keywords**: 带同义词和口语术语的扩展关键词（5-10项）
  - **domain_context**: 用于生成的查询类型和域特定模板
